> This repository is **not** a full dump of the private workspace. It is a deliberately filtered, low-risk, GitHub-safe subset prepared with a whitelist publishing strategy.

---

## What this repo is

This repository collects a small public slice of a larger OpenClaw working environment.

It currently focuses on:

- selected skill definitions and helper scripts
- reusable documentation
- lightweight local automation that does not require private data to understand

The goal is to publish useful, portable pieces without exposing private notes, runtime state, credentials, large local datasets, or proprietary dependencies.

## Highlight: Obsidian daily summary helper

`skills/obsidian/daily_summary.py` restores a lightweight daily-summary workflow for an Obsidian vault.

Current behavior:
- reads from `memory/YYYY-MM-DD.md` or accepts direct summary text
- generates `obsidian/每日总结/YYYY-MM-DD.md`
- avoids dependence on desktop Obsidian or `obsidian-cli` for server-side cron usage

Typical usage:

```bash
python3 skills/obsidian/daily_summary.py
```

Generate a specific day:

```bash
python3 skills/obsidian/daily_summary.py --date 2026-03-24 --force
```

Generate from direct text input:

```bash
python3 skills/obsidian/daily_summary.py --date 2026-03-25 --summary-text "your summary here" --force
```

## Publishing policy

This repository uses a **whitelist-first** publishing policy.

That means only selected low-risk content is published.

### Intentionally excluded

The following categories are intentionally kept out of GitHub:

- secrets and local credentials (`.env`, keys, tokens)
- runtime state (`logs/`, `*.pid`, caches)
- generated outputs and artifacts
- large local models
- proprietary SDK binaries
- local data snapshots, DuckDB, Parquet, backtest data
- personal memory files and private Obsidian vault content
- backup directories and temporary work products

---

## Why the repo is intentionally small

The original workspace contains many things that are useful locally but inappropriate for a public repository, for example:

- data-heavy trading research assets
- environment-specific automation
- output files generated by publishing workflows
- private operational notes and memory files

Keeping the public repo small makes it:

- safer to publish
- easier to understand
- easier to maintain
- less likely to leak unrelated local context

---

## Notes for reuse

Some skills in the private workspace depend on:

- local filesystem layout
- external credentials
- proprietary APIs or SDKs
- machine-specific runtime assumptions

So even when a skill is published here, you may still need to adapt paths or provide your own environment/configuration before using it directly.
